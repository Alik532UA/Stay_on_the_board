## 🏗️ Архітектура даних (Firebase)

### 1. Firestore (Збереження стану)
Колекція `rooms`. Кожен документ — це кімната.
```typescript
interface Room {
  id: string;             // Унікальний ID (код кімнати)
  name: string;           // Назва ("Кімната гравця Alex" або кастомна)
  hostId: string;         // ID гравця-хоста (власника)
  status: 'waiting' | 'playing' | 'finished';
  createdAt: number;
  lastActivity: number;   // Для очищення старих кімнат
  isPrivate: boolean;     // Чи відображати в списку
  settingsLocked: boolean;// Чи можуть інші змінювати налаштування (Варіант 4A->B)
  
  // Ігровий стан
  gameState: SyncableGameState | null;
  
  // Гравці (масив або мапа)
  players: {
    [playerId: string]: {
      name: string;
      color: string;
      isReady: boolean;   // Для старту (Варіант 5C)
      joinedAt: number;   // Для міграції хоста
      isOnline: boolean;  // Статус з'єднання
    }
  };
  
  // Налаштування гри (синхронізуються в лобі)
  settings: GameSettingsState;
}
```

### 2. Realtime Database (Presence & Ping)
Використовується для миттєвого відстеження статусу "онлайн/офлайн" та міграції хоста.
*   Шлях: `/status/{roomId}/{playerId}` -> `online` / `offline` / `last_seen`.

## 🛠 Технічні деталі реалізації "Лінивого очищення"

У `RoomService.getPublicRooms()`:
1.  Отримати список кімнат з Firestore.
2.  Для кожної кімнати:
    *   Якщо `(Date.now() - room.lastActivity) > 2 хвилини` (або 10 сек для тесту):
        *   Викликати `deleteDoc(roomRef)`.
        *   Не додавати цю кімнату в список для відображення.
3.  Повернути відфільтрований список.

*Примітка: Це означає, що перший користувач, який зайде в меню, трохи "попрацює прибиральником", але для інших список буде чистим.*

## 🛡 Безпека (Базові правила Firestore)

На етапі MVP ми покладаємось на клієнтську валідацію (1A), але додамо базові правила безпеки в Firestore Rules:
1.  Тільки учасники кімнати можуть писати в неї.
2.  Не можна змінювати чужий `playerId`.
3.  Не можна змінювати історію ходів (тільки додавати).
