### **Загальний аналіз та стратегія**

Проєкт перебуває у фазі активного рефакторингу від старої архітектури (`index-old-architecture.html`) до нової, що базується на `state-manager.js`, `view-manager.js` та компонентному підході (`*-component.js`). Багато з виявлених проблем є наслідком цього перехідного періоду, зокрема неправильне управління життєвим циклом компонентів та побічні ефекти від зміни стану. Моя стратегія полягає у тому, щоб кожен фікс не лише вирішував проблему, а й посилював нову архітектуру, забезпечуючи кращу інкапсуляцію та передбачуваність.

### **Bug Report #12: Некоректне відображення заблокованих клітинок**

**Аналіз та першопричина:**
Проблема полягає у функції `renderBoard` в `js/components/game-board-component.js`. Логіка рендерингу клітинок не є взаємовиключною. Спочатку код може правильно додати клас `blocked-cell` та символ `✗`. Однак, якщо ця ж клітинка також потрапляє під умову `isAvailable` або `isHighlighted`, наступний блок коду перезаписує `innerHTML` клітинки, видаляючи символ блокування. Заблокована клітинка повинна мати найвищий пріоритет у логіці рендерингу.

**Стратегічний план для виправлення:**

1.  **Файл для модифікації:** `js/components/game-board-component.js`
2.  **Функція:** `renderBoard`
3.  **Дія:** Змінити логіку всередині циклу `for (let col = 0; col < boardSize; col++)`. Потрібно забезпечити, щоб перевірка на `isBlocked` була першою і остаточною для даної клітинки.

    **Замінити цей блок коду:**
    ```javascript
    // --- Відображення заблокованих клітинок ---
    const isBlocked = blockedCells.some(c => c.row === row && c.col === col);
    if (isBlocked) {
        cell.classList.add('blocked-cell');
        cell.innerHTML = '<span class="blocked-x">✗</span>';
        Logger.debug('[GameBoardComponent] renderBoard: blocked cell rendered', { row, col });
    } else {
        const cellValue = board[row][col];
        // --- Фігура ---
        if (cellValue === 1) {
            cell.classList.add('player-piece');
            cell.innerHTML = '<span class="crown">👑</span>';
        } else if (cellValue === 2) {
            cell.classList.add('player-piece', 'player2');
            cell.innerHTML = '<span class="crown">👑</span>';
        } else {
            cell.classList.add('empty-cell');
            // --- Доступні ходи ---
            const isHighlighted = highlightedMoves.some(move => move.newRow === row && move.newCol === col);
            const isAvailable = availableMoves.some(move => move.newRow === row && move.newCol === col);
            if (isHighlighted) {
                cell.classList.add('highlighted-move');
                cell.innerHTML = '<span class="move-dot move-dot-highlighted"></span>';
            } else if (isAvailable && showingAvailableMoves) {
                cell.classList.add('available-move');
                cell.innerHTML = '<span class="move-dot"></span>';
            } else {
                cell.innerHTML = '';
            }
        }
    }
    gameBoard.appendChild(cell);
    ```

    **На цей покращений блок:**
    ```javascript
    const isBlocked = blockedCells.some(c => c.row === row && c.col === col);

    if (isBlocked) {
        cell.classList.add('blocked-cell');
        cell.innerHTML = '<span class="blocked-x">✗</span>';
        Logger.debug('[GameBoardComponent] renderBoard: blocked cell rendered', { row, col });
    } else {
        const cellValue = board[row][col];
        if (cellValue === 1) {
            cell.classList.add('player-piece');
            cell.innerHTML = '<span class="crown">👑</span>';
        } else if (cellValue === 2) {
            cell.classList.add('player-piece', 'player2');
            cell.innerHTML = '<span class="crown">👑</span>';
        } else {
            cell.classList.add('empty-cell');
            const isHighlighted = highlightedMoves.some(move => move.newRow === row && move.newCol === col);
            const isAvailable = availableMoves.some(move => move.newRow === row && move.newCol === col);

            if (isHighlighted) {
                cell.classList.add('highlighted-move');
                cell.innerHTML = '<span class="move-dot move-dot-highlighted"></span>';
            } else if (isAvailable && showingAvailableMoves) {
                cell.classList.add('available-move');
                cell.innerHTML = '<span class="move-dot"></span>';
            } else {
                cell.innerHTML = '';
            }
        }
    }
    gameBoard.appendChild(cell);
    ```

**Обґрунтування:**
Ця зміна реструктурує логіку, роблячи її більш надійною. Перевірка на `isBlocked` тепер є першочерговою. Якщо клітинка заблокована, вона отримує відповідний стиль та контент, і подальші перевірки на фігури чи доступні ходи для неї не виконуються. Це усуває можливість перезапису `innerHTML` і гарантує коректне відображення.

**Верифікація:**
1.  Запустити гру.
2.  Увімкнути "Режим заблокованих клітинок".
3.  Зробити хід.
4.  Переконатися, що клітинка, з якої було зроблено хід, стала чорною з червоним символом `✗` і не містить інших позначок.

### **Bug Report #13: Відстань за замовчуванням не скидається при зміні розміру дошки**

**Аналіз та першопричина:**
Проблема виникає у `js/components/game-board-component.js` в обробнику події `change` для селектора `#board-size-select`. Код змінює `game.boardSize` у `stateManager`, але не враховує, що поточне значення `game.selectedDistance` може стати невалідним для нового розміру дошки (наприклад, відстань 4 на дошці 3x3). Це призводить до неузгодженості стану.

**Стратегічний план для виправлення:**

1.  **Файл для модифікації:** `js/components/game-controls-component.js`
2.  **Функція:** `subscribeToState`
3.  **Дія:** Додати логіку для скидання відстані при зміні розміру дошки. Це більш архітектурно правильно, оскільки компонент контролів повинен сам реагувати на зміни, що впливають на його стан.

    **Додати цей блок коду всередину `subscribeToState`:**
    ```javascript
    // Підписка на зміни розміру дошки
    this.subscribe('game.boardSize', (newSize) => {
        Logger.debug('[GameControlsComponent] Board size changed to:', newSize);
        this.generateDistanceButtons(newSize); // Оновлюємо лише кнопки дистанції
        this.bindEvents(); // Додаємо обробники на нові кнопки
        
        // Скидаємо відстань до 1 при зміні розміру дошки, якщо поточна відстань невалідна
        const currentDistance = stateManager.getState('game.selectedDistance');
        if (currentDistance !== null && currentDistance >= newSize) {
            stateManager.setState('game.selectedDistance', 1);
            Logger.debug('[GameControlsComponent] Reset distance to 1 due to board size change:', { 
                oldDistance: currentDistance, 
                newBoardSize: newSize, 
                maxDistance: newSize - 1 
            });
        }
    });

    // Підписка на зміни selectedDistance для діагностики
    this.subscribe('game.selectedDistance', (distance) => {
        Logger.debug('[GameControlsComponent] game.selectedDistance changed:', { 
            distance, 
            boardSize: stateManager.getState('game.boardSize'),
            maxDistance: (stateManager.getState('game.boardSize') || 3) - 1
        });
    });
    ```

**Обґрунтування:**
Перенесення цієї логіки в `GameControlsComponent` покращує інкапсуляцію. Компонент, що відповідає за вибір відстані, тепер сам реагує на зміну розміру дошки і підтримує свій стан у консистентному вигляді. Це відповідає принципам реактивного програмування та робить систему більш надійною.

**Верифікація:**
1.  Запустити гру на дошці 5x5.
2.  Вибрати напрямок і відстань 4.
3.  Змінити розмір дошки на 3x3.
4.  Переконатися, що вибрана відстань автоматично скинулася на 1, а не залишилася 4.

### **Bug Report #15: Дублювання обробників подій у головному меню**

**Аналіз та першопричина:**
Проблема в `js/components/main-menu-component.js`. Метод `render` викликається кожного разу при переході на головне меню, і він, у свою чергу, викликає `bindEvents`. Попередні обробники подій не видаляються, що призводить до їх накопичення та багаторазового спрацьовування. Використання прапорця `eventsBound` є поширеним, але крихким рішенням.

**Стратегічний план для виправлення:**

1.  **Файл для модифікації:** `js/components/main-menu-component.js`
2.  **Функція:** `bindEvents`
3.  **Дія:** Перед додаванням нових обробників необхідно гарантовано видалити всі старі. Найпростіший і найнадійніший спосіб для цього — клонування та заміна вузлів кнопок.

    **Замінити цей блок коду:**
    ```javascript
    bindEvents() {
        // ... (старий код з прапорцем eventsBound)
    }
    ```

    **На цей покращений блок:**
    ```javascript
    bindEvents() {
        // --- ДОДАНО: очищення попередніх обробників ---
        this.element.querySelectorAll('button').forEach(btn => {
            btn.replaceWith(btn.cloneNode(true));
        });

        this.element.querySelector('#btn-vs-computer')?.addEventListener('click', () => {
            stateManager.navigateTo('gameBoard', { gameMode: 'vsComputer' });
        });
        this.element.querySelector('#btn-local-game')?.addEventListener('click', () => {
            stateManager.navigateTo('gameBoard', { gameMode: 'local' });
        });
        this.element.querySelector('#btn-online')?.addEventListener('click', () => {
            window.location.href = 'peer-mvp.html';
        });
        // ... (решта обробників без змін) ...
    }
    ```
4.  **Дія 2:** Видалити прапорець `this.eventsBound` з конструктора та будь-яких інших місць у компоненті.

**Обґрунтування:**
Метод `node.replaceWith(node.cloneNode(true))` є ідіоматичним та надійним способом видалення всіх обробників подій з елемента, оскільки клонований вузол не успадковує їх. Це гарантує, що при кожному виклику `bindEvents` ми працюємо з "чистими" елементами, що повністю усуває проблему дублювання. Цей підхід простіший і надійніший за ручне відстеження та видалення кожного обробника.

**Верифікація:**
1.  Запустити гру, перейти в налаштування.
2.  Повернутися в головне меню.
3.  Перейти в правила.

### **Bug Report #17: Дублювання рендерингу GameControlsComponent**

**Аналіз та першопричина:**
Проблема в `js/components/game-controls-component.js`. Метод `render` не має захисту від повторного виклику. Оскільки `GameBoardComponent` може перерендеритися (наприклад, при зміні розміру дошки) і викликати рендеринг `GameControlsComponent` знову, це призводить до дублювання елементів керування.

**Стратегічний план для виправлення:**

1.  **Файл для модифікації:** `js/components/game-controls-component.js`
2.  **Функція:** `render`
3.  **Дія:** Додати прапорець `this.isRendered` для захисту від повторного рендерингу.

    **Додати на початку методу `render`:**
    ```javascript
    render() {
        // Перевіряємо, чи вже рендерилися
        if (this.isRendered) {
            Logger.debug('[GameControlsComponent] Already rendered, skipping');
            return;
        }
        
        Logger.info('[GameControlsComponent] render: початок');
        // ... (решта коду методу)
    ```
4.  **Дія 2:** Встановити прапорець в `true` в кінці методу `render`.

    **Додати в кінці методу `render`:**
    ```javascript
        // ... (код рендерингу)
        this.isRendered = true;
        Logger.info('[GameControlsComponent] render: завершено');
    }
    ```
5.  **Дія 3:** Додати скидання прапорця в метод `destroy`.

    **Додати в метод `destroy`:**
    ```javascript
    destroy() {
        // ... (інший код очищення)
        this.isRendered = false;
        super.destroy();
    }
    ```

**Обґрунтування:**
Додавання прапорця `isRendered` є простою та ефективною реалізацією ідемпотентності для методу `render`. Це гарантує, що вміст компонента буде створено лише один раз за його життєвий цикл, навіть якщо `render` буде викликано помилково кілька разів. Це робить компонент більш стійким до непередбачуваних викликів з боку батьківських компонентів.

**Верифікація:**
1.  Запустити гру, перейти на ігровий екран.
2.  Змінити розмір дошки кілька разів.
3.  Переконатися, що панель керування (кнопки напрямку, відстані) не дублюється і залишається в одному екземплярі.
4.  Перевірити логи консолі на наявність повідомлення `[GameControlsComponent] Already rendered, skipping`.
